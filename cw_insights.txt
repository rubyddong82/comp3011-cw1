The system runs as follows:

1. main.py launches server/server_main.py first.
2. server_main.py initializes the system and database.
3. Once the server is ready, main.py launches client/client_main.py.
4. The client communicates with the server through HTTP requests.

The server manages a local SQL database containing datasets.

Initially the server loads a predefined dataset into the database (for example a movie dataset). This dataset is used for analytics and row evaluation.

Later the system may fetch datasets from external sources (Kaggle, UCI, etc.) through server_client.py, but that is not the focus right now.

The client provides an interactive CLI interface that allows the user to explore and manipulate the dataset.

---

OVERALL PROGRAM GOAL

The program allows users to:

• Search rows
• Filter rows
• Perform CRUD operations on rows
• View row-level evaluation metrics
• Perform column analysis and dataset analytics
• Customize schema by adding user-defined columns

The system is intended to behave like a lightweight dataset exploration and analytics tool.

---

DATABASE RESPONSIBILITIES (db.py)

The database module (db.py) handles all data logic.

db.py is responsible for three main categories of functionality:

1. Row operations and row evaluation
2. Column analysis (dataset analytics)
3. Schema customization

The CLI and server logic should not implement database logic directly. They should call functions inside db.py.

---

ROW WORKFLOW

Row workflows include the following operations:

CREATE ROW
• Fetch the table schema (column list)
• Prompt the user to fill values for each column
• Insert the new row into the database
• If row evaluation exists, compute evaluation metrics for the new row

READ ROW
• Perform search and filter to locate candidate rows
• Display candidate rows
• Candidates may be sorted by evaluation score
• User selects a row
• Show full row information
• Show row evaluation metrics (how the row compares to the dataset)

UPDATE ROW
• Search/filter to locate candidate rows
• Optionally sort candidates by evaluation score
• User selects a row
• Display the row information
• Show row evaluation
• User chooses which column to modify
• Update the row
• Recalculate evaluation metrics

DELETE ROW
• Search/filter to locate candidate rows
• Display candidate rows
• User selects a row
• Show row information and evaluation
• Confirm deletion
• Delete the row

SEARCH AND FILTER DESIGN

Search and filter are combined mechanisms used to generate candidate rows.

SEARCH
• Search operates on one manually selected "name column"
• Example: title for movie datasets
• User types a search value
• Search performs text matching on that column

FILTER
• Filters operate on all columns except:
  - the search/name column
  - row index or internal ID

• User can select one or more columns to filter
• For each filter column, the user can:
  - choose from candidate values
  - type a value manually

FILTER VALUE CANDIDATES
• Filter value options are sorted by frequency
• Example:

Drama (3400)
Comedy (2800)
Action (2500)

This helps users choose the most common categories easily.

CANDIDATE ROW GENERATION

Candidate rows are determined using both search and filter conditions together:

• If only search exists → use search results
• If only filters exist → use filtered results
• If both exist → apply both conditions

Candidates are displayed to the user.

If row evaluation exists and evaluation sorting is enabled, candidate rows should be sorted by evaluation score.

Otherwise fallback ordering is used.

Evaluation sorting should be a toggle option but should default to ON if evaluation exists.

---

ROW EVALUATION

Each row can have an evaluation score based on dataset-wide metrics.

Row evaluation measures how a row compares to the rest of the dataset.

Example metrics for a movie dataset:

• rating percentile
• genre ranking
• runtime compared to average runtime
• popularity metrics
• composite evaluation score

Row evaluation should be computed dynamically using dataset statistics.

Row evaluation results should be displayed when reading or updating rows.

Candidate search results can also be sorted using this evaluation score.

Column analysis allows the user to analyze dataset statistics.

Column analysis has two categories:

1. Default analysis
2. Custom analysis

DEFAULT ANALYSIS

Default analysis is split into two types:

A. Mono-column analysis
B. Cross-column analysis

---

MONO-COLUMN ANALYSIS

Mono-column analysis analyzes a single column.

There are two cases:

Categorical columns

Example:
genre

Output:
percentage distribution of labels

Example:

Drama 34%
Comedy 28%
Action 21%
Thriller 17%

Numeric columns

Example:
rating, runtime, revenue

Steps:

1. Determine minimum and maximum values
2. Divide the range into 5 buckets
3. Calculate percentage distribution

Example:

0–2 → 1%
2–4 → 8%
4–6 → 25%
6–8 → 48%
8–10 → 18%

This forms a histogram.

---

CROSS-COLUMN ANALYSIS

Cross-column analysis uses two columns:

Independent column (grouping column)
Dependent column (measured column)

Examples:

genre → average rating

year → most common genre

If dependent column is categorical:
• show most frequent label per independent group

If dependent column is numeric:
• compute average value per independent group

Example:

Genre → Average Rating

Drama → 7.8
Action → 7.1
Comedy → 6.4

---

CUSTOM ANALYSIS

Custom analysis allows users to define their own analytics configuration.

Users choose:

• independent column
• dependent column
• aggregation type
• optional bucket count

Examples:

average revenue by genre
median rating by year
movie count by language
runtime distribution with 10 buckets

Custom analysis generalizes the default analysis system.

SCHEMA CUSTOMIZATION

The system allows schema customization.

Users can add custom columns to the dataset to store personal metadata.

Examples of custom columns:

• favourite

Schema customization operations include:

• delete column

Once a column exists, users can populate values using normal row update workflows.

Example:

Add column: favourite

Then rows can store:

movie → favourite = true

This allows personal dataset annotations.
